## üîÑ Updating Documentation

### How to Suggest Updates

1. **Small Changes** (typos, clarifications):
   - Create PR directly
   - Tag relevant team members

2. **Standard Changes** (new patterns, rules):
   - Discuss in team meeting first
   - Get consensus
   - Create PR with changes
   - Update date and version number

3. **Major Changes** (restructuring, new documents):
   - Write proposal
   - Present to team
   - Get approval
   - Implement changes
   - Announce to all teams

### Documentation Versioning

Format: `vMAJOR.MINOR.PATCH`

- **MAJOR**: Complete restructure or major changes
- **MINOR**: New sections or significant updates
- **PATCH**: Small corrections and clarifications

Current Version: **v1.0.0** (November 19, 2025)

## üÜò Common Questions

**Q: Which document should I read first?**
A: Start with CODING_STANDARDS.md, then QUICK_REFERENCE.md

**Q: Do I need to memorize all standards?**
A: No, use QUICK_REFERENCE.md as your daily guide. Understanding > Memorization

**Q: Can I use my own coding style?**
A: Please follow team standards for consistency. Suggest improvements if you have better ideas.

**Q: What if AI generates code that doesn't match our standards?**
A: Review and modify the code. Update the prompt in AI_PROMPTS.md to get better results.

**Q: How do I know which AI prompt to use?**
A: Check the table of contents in AI_PROMPTS.md and pick the one matching your task.

**Q: What if I disagree with a standard?**
A: Discuss in team meeting, propose alternative, update if team agrees.

**Q: How often are these documents updated?**
A: As needed. Check commit history for recent changes.

**Q: Can I create new documentation?**
A: Yes! Propose it to the team, get approval, create PR.
